# 评论互动、好友与聊天设计

## 目标

为文章评论补充作者删除、共用评论输入框、评论点赞、举报处理和账号角色标识；增加公开资料气泡、好友申请以及好友之间的一对一实时聊天。

## 评论规则

- 用户只能删除自己的评论和回复，管理员保留内容管理权限。
- 删除采用软删除。存在有效回复时保留“该评论已删除”的上下文占位。
- 评论和回复共用文章底部输入框，回复状态明确显示目标账号并允许取消。
- 点赞按用户和评论唯一记录；举报按举报人和评论唯一记录，可重新提交已处理的举报。
- 管理员可以查看待处理举报、定位原评论，并执行屏蔽、删除或驳回。

## 好友与公开资料

- 公开资料只包含头像、昵称、用户名、个人介绍、角色和加入时间，不返回邮箱、IP 或登录设备。
- 好友关系使用标准化用户编号对，避免同一对用户产生重复关系。
- 好友申请需要对方接受；只有已接受的好友可以建立会话。
- 好友申请可以填写不超过 120 字的备注。接受或拒绝申请时，原申请通知会在同一事务中标记为已读，并给申请人生成持久化系统通知。

## 聊天架构

- 历史消息通过 HTTP 分页读取并持久化到 MySQL。
- 实时新消息和已读回执通过同域名 `/socket.io` WebSocket 传输，复用 HTTPS 443 端口。
- Redis 只保存带过期时间的在线状态，不保存聊天历史。
- 每个账号最多保持三个聊天连接；单条消息最多 2000 字符；单连接十秒最多发送十二条消息。
- 聊天组件挂载在根布局中。桌面端使用右下角悬浮窗，移动端使用接近全屏的底部面板；切换页面不会中断 Socket、当前会话或未发送草稿。
- `/messages` 仅作为旧链接兼容入口，负责打开悬浮聊天窗，不再提供独立消息页面。

## 附件与通知

- 单条消息最多包含 9 个图片或文件；图片单张不超过 8MB，普通文件单个不超过 20MB，总计不超过 50MB。
- 图片支持 JPG、PNG、WebP；普通文件使用明确的文档与压缩包白名单。可执行文件和脚本被拒绝，服务端校验扩展名与文件内容特征。
- 选择、粘贴或拖入的文件先保留在浏览器本地，点击发送时才通过 HTTP 上传；Socket 消息只传递已上传附件编号。
- 附件下载必须携带登录令牌，并校验当前账号属于对应会话。尚未发送的附件仅上传者可访问。
- 好友申请结果和举报处理结果写入 MySQL 通知表。顶部消息图标和悬浮聊天窗展示未读通知；管理员待办图标展示未处理举报并可定位原评论。

## 资源边界

当前服务器为 2 vCPU、2 GiB。第一版以几十名并发在线用户为目标。消息历史查询分页、连接数限制和发送频率限制用于控制 API 内存、MySQL 写入和网络缓冲压力。
