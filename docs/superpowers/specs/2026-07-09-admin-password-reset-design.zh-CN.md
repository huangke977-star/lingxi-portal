# 超级管理员修改用户密码设计

日期：2026-07-09

## 目标

在现有用户管理后台中加入“修改密码”能力，让 `isSuperAdmin = true` 的超级管理员可以为任意账号设置新密码，同时确认后台不提供删除用户入口。

## 范围

本阶段包含：

- 后端新增 `PATCH /users/:id/password` 接口。
- 接口继续使用 `JwtAuthGuard` 和 `SuperAdminGuard`，只有超级管理员可调用。
- 新密码必须通过 DTO 校验，最短 8 位。
- 新密码必须使用现有 `PasswordService` 哈希后写入 `passwordHash`。
- 响应只返回用户公开信息，不返回密码哈希。
- `/admin` 用户表格新增“修改密码”按钮和确认弹窗。
- 前端提交前校验两次输入一致。
- 保持没有删除 API，也没有删除按钮。

本阶段不包含：

- 普通用户自助修改自己的密码。
- 找回密码、邮箱验证码或二次验证。
- 删除用户、批量删除用户或软删除。
- 角色 CRUD。

## 设计

后端沿用现有 users 管理模块，不新增独立密码管理模块。`UsersController` 增加 `PATCH /users/:id/password`，请求体为 `{ "password": "new-password" }`。`UsersService` 注入 `PasswordService`，先哈希新密码，再更新目标用户的 `passwordHash`，最后按 `AuthenticatedUser` 结构返回用户摘要。

前端 `/admin` 在每个用户行的操作区增加“修改密码”按钮。点击后打开同页弹窗，显示目标账号，要求输入新密码和确认密码。保存成功后关闭弹窗并显示成功消息；失败时只显示错误消息，不保留或回显密码。

“删除用户”保持不可用：本次不创建删除接口，不加入删除按钮，也不在 UI 上保留隐藏删除入口。

## 权限与安全

- 只有 `isSuperAdmin = true` 的账号可以调用改密码接口。
- 普通管理员角色 `administrator` 如果不是超级管理员，也不能调用接口。
- 密码不会写入日志、页面状态提示、文档或对话。
- 后端不信任前端隐藏按钮，权限必须由 guard 强制执行。
- 响应体不得包含 `passwordHash`。

## 验收标准

- 超级管理员可以在后台为任意账号设置新密码。
- 非超级管理员调用 `PATCH /users/:id/password` 返回 403。
- 少于 8 位的新密码返回 400。
- 改密码接口响应不包含 `passwordHash`。
- `/admin` 页面没有删除按钮，且有“修改密码”按钮。
- API 测试、API lint、Web lint 和生产构建通过。
